ollim-bot is developed continuously without formal version releases.
This changelog tracks major features and milestones in reverse
chronological order.

<Note>ollim-bot has been in active development since February 9, 2026 with 588 commits across 35 active development days.</Note>

## Apr 19-21, 2026

**Auto-memory disabled, reminder and webhook reliability fixes**

- **`autoMemoryEnabled: false` by default** — bootstrap now writes `autoMemoryEnabled: false` to `~/.ollim-bot/.claude/settings.json` unless you set the key explicitly. ollim-bot persists context through its own markdown files ([`IDENTITY.md`](/personalizing/personality), [`USER.md`](/personalizing/user-context), reflections), so Claude Code's auto-memory would write a parallel note track the bot never reads.
- **Reminder file cleanup on dispatch failure** — if `validate_dispatch` fails at fire time for a one-shot reminder, the orphaned `.md` is now removed instead of being left behind. Previously the `DateTrigger` job would fire once, APScheduler would drop it, and `sync_all`'s "already registered" guard would silently block re-registration — killing the reminder forever.
- **Webhook `AppKey` migration** — `_bg_tasks` now uses a typed `web.AppKey` instead of a raw string key, silencing aiohttp 3.9+'s `NotAppKeyWarning` on every webhook POST.

## Apr 7-21, 2026

**Multi-calendar support and configurable task list**

- **Multi-calendar queries** — `ollim-bot cal today` and `cal upcoming` now query every calendar listed in the new `google_calendars` config key (comma-separated IDs, default `primary`). Events are merged across calendars and sorted client-side, and each line is tagged with its source calendar ID when more than one calendar is configured. A 404 on an individual calendar is logged and skipped. See [Google Calendar](/integrations/google-calendar#configuring-which-calendars-to-query).
- **`ollim-bot cal calendars`** — new subcommand that lists every calendar your Google account can access, showing the ID and display name. Use it to discover IDs for `google_calendars` or `--calendar`.
- **`--calendar` flag on every `cal` subcommand** — read commands use it to query a single calendar, write commands (`add`, `show`, `update`, `delete`) use it to target a non-default calendar. Defaults to the first configured calendar for writes.
- **Calendar delete buttons encode the calendar ID** — the `event_del` button action now accepts `event_del:<calendar_id>/<event_id>` in addition to the legacy `event_del:<event_id>` (defaults to `primary`), so events on non-primary calendars are deleted from the correct calendar.
- **Configurable task list** — new `google_task_list` config key sets the default Google Tasks list (default `@default`). Every `ollim-bot tasks` subcommand accepts `--list <id>` to override per invocation. Task-complete and task-delete buttons honor the configured list and can encode a specific list ID into the button action.
- **Degenerate-config fallback** — an empty or whitespace-only `google_calendars` value falls back to `primary` instead of crashing, keeping `cal calendars` runnable for recovery.

## Mar 31 – Apr 21, 2026

**Counterfactual trajectory testing**

- **`counterfactual` CLI** — new top-level command (a sibling of `ollim-bot`,
  not a subcommand) that replays a real production transcript from a chosen
  user-message UUID, applies an intervention (system prompt append/replace,
  tool disallow, model swap, or alternate message), and prints the new
  response next to the original. Installed alongside `ollim-bot` by
  `uv tool install --editable .`. See
  [CLI reference](/development/cli-reference#counterfactual) and
  [Counterfactual trajectory testing](/development/testing#counterfactual-trajectory-testing).
- **`--with-baseline`** — optional second fork that re-runs the original
  settings so differences between baseline and variant isolate the
  intervention's effect from sampling noise. Doubles cost.
- **`src/ollim_bot/eval/` sub-package** — module layout for the
  counterfactual implementation, with unit tests covering session truncation,
  UUID resolution, and original-response extraction.
- **`counterfactual-test` Claude Code skill** — bundled at
  `.claude/skills/counterfactual-test/SKILL.md` in the source repo. Invoke
  with `/counterfactual-test`; model invocation is disabled. This is a
  Claude Code dev-harness skill, separate from
  [ollim-bot runtime skills](/extending/skills).
- **UUID prefix matching** — the `rewind_uuid` argument accepts prefixes and
  resolves them against user-message UUIDs only. Assistant and tool-result
  UUIDs are rejected with a clear error, and the first record in a session
  cannot be a rewind point.

## Apr 13, 2026

**Agent tool usage guardrail**

- **Never use `run_in_background` for `Bash`** — the `claude_code` preset encourages background execution for long-running commands, but background results are never delivered back in the SDK streaming context, so commands silently lose their output. The system prompt now overrides this with an explicit "never" instruction.

## Apr 9, 2026

**Structural reflections for background forks**

- **Execution traces after every background fork** — when a [background fork](/scheduling/background-forks#reflections) finishes (success, timeout, or failure), a Haiku meta-fork writes a short trace to `~/.ollim-bot/reflections/<item-id>/<timestamp>.md`. Each trace records the job status, whether `report_updates` was called, any error info, and a 1-3 sentence summary of what the task was meant to do.
- **Pre-filled deterministic fields** — Status, Report filed, and Errors are filled in Python from the fork outcome; Haiku only generates the **Trace** line. This prevents hallucinated statuses and literal copying of placeholder text.
- **Isolated write-only scope** — the meta-fork runs with `model: haiku`, thinking off, and a single `Write(reflections/**/*.md)` tool. It runs in the parent fork's `finally` block after contextvars reset, with a 60-second timeout. Reflection failures (including cancellation) are logged and swallowed — they never affect the original job.
- **Per-job opt-out** — set `reflect: false` in a routine, reminder, or webhook's YAML frontmatter. Default is `true`.

## Apr 6-7, 2026

**Fork turn-count, skill invocation, and permission label fixes**

- **Fork turn-count bump on all entry paths** — `/fork`, the `enter_fork` MCP tool, and idle-timeout prompts now call `bump_fork_turn()`. Previously these paths never bumped at all, leaving `turn_count` at 0 and causing `exit_fork` and `report_updates` to reject with "cannot exit on the first turn" even after multiple turns. Completes the [Mar 25](#mar-25-2026) fix that reordered the bump to run before dispatch.
- **Routine re-registration on any field change** — `_registered_routines` now stores the full `Routine` dataclass instead of just the cron string, so changes to `model`, `allowed_tools`, `message`, `thinking`, and other fields are detected and re-registered automatically without a restart. Previously only cron changes triggered re-registration.
- **`/skill-name` invocation restored** — routines, reminders, and webhooks that start with `/skill-name` now invoke the skill as expected. `append_context()` places the timestamp and pending-update preamble *after* the message body so the `/command` stays at position 0 for CLI skill recognition.
- **Deferred labels skipped for interactively surfaced permissions** — tools that go through interactive Discord approval no longer render a duplicate label at the bottom of the response with the wrong hint (`use /permissions default`). The previous render duplicated info the user had already seen.
- Internal: consolidated `clear_denied`, `clear_errored`, and `clear_surfaced` into a single `clear_label_state` helper for coherent label-state transitions.

## Mar 31, 2026

**`ollim-bot chat` — Discord-free agent interface**

- **`ollim-bot chat` CLI** — new subcommand for talking to the agent from
  the terminal without Discord. Uses a `ChatChannel` duck type that captures
  MCP tool output to stdout. Supports a `--model` flag for switching between
  Claude and local Ollama models, and runs with
  `permission_mode="bypassPermissions"` so the terminal session never prompts
  for tool approvals. See
  [CLI reference](/development/cli-reference#chat).
- **Shared main session** — `chat` resumes the existing **main session** if
  one is present, so conversations persist between Discord and the terminal
  in both directions.
- **Clean shutdown** — exiting chat (Ctrl-D or Ctrl-C) disconnects the SDK
  client via a new public `Agent.close()` method so the session transcript
  is flushed to disk before the process exits.
- Internal: `check_auth()` extracted to `auth.py` for reuse between the
  Discord startup path and the chat entry point.

**Routine validator hook, `/improve-routine` skill, and reply-to-fork tracking**

- **`routine_validator` PreToolUse hook** — a new built-in SDK hook validates routine `.md` files on `Write` and `Edit`. Blocks on missing `id`, missing `cron`, invalid cron (must be 5 fields), unclosed frontmatter, and malformed YAML. Warns on unscoped `Bash`/`Write`/`Edit` in `allowed-tools`, delegation combined with unscoped writes, underscore keys where canonical form is hyphenated, unknown frontmatter keys, and routines over 200 lines. See [Hooks](/extending/hooks#routine-validator) and [Routines](/scheduling/routines#validation-on-write).
- **`/improve-routine` bundled skill** — new bundled [skill](/extending/skills#bundled-skills) that diagnoses routine quality issues — data source confusion, embellishment propagation, model/tool mismatches, report quality. Runs in plan mode, presents a failure report for user sign-off, then applies fixes and spawns two critic subagents (completeness + regression) to verify each change prevents the reported failure.
- **Fork exit embed tracking** — "Fork Ended — summary queued" embeds are now written to `fork_messages.json` via `track_fork_message()` across all five exit paths (`bot.py`, `views.py` ×3, `scheduler.py`). Previously, replying to a fork-end embed did not resume the fork session.
- **Idle-check and `save_context` message tracking** — the idle-check status message and `save_context` confirmation embed now call `track_message()`, so replying to them is recognized by reply-to-fork lookup.

## Mar 29-31, 2026

**Docker deployment**

- **Dockerfile and compose stack** — the repo now ships `Dockerfile`, `docker-compose.yml`, and `docker-entrypoint.sh`. `docker compose up -d` brings up ollim-bot alongside a local [Ollama](https://ollama.com) server, with named volumes (`ollama_data`, `bot_data`, `cli_data`) for persistence across rebuilds. See [Docker deployment](/self-hosting/guide#docker-deployment).
- **Auto-pull Ollama models** — the entrypoint checks whether `ANTHROPIC_MODEL` exists on the Ollama server and pulls it automatically if not, so the first start downloads the model before the bot launches.
- **Skip OAuth when `ANTHROPIC_AUTH_TOKEN` is set** — `is_authenticated()` is no longer called when the token is present, fixing a crash in restricted container environments where the bundled CLI can't write temp files.
- **Non-root container user** — the bot runs as an unprivileged `bot` user inside the container. Ollama's port is bound to `127.0.0.1` on the host, not exposed to the network.
- Internal: `--no-dev` in the Dockerfile `CMD` skips dev dependency downloads, session ID reset in chat mode prevents stale state across container restarts.

## Mar 28, 2026

**Semver releases, `/version` command, and tag-based auto-update**

- **`/version` slash command** — shows the running bot version. Prints `ollim-bot v0.2.0` when `HEAD` is on a tag, or `ollim-bot v0.1.0+dev (abc1234)` for dev builds ahead of the nearest tag. Derived from `git describe --tags`, with an `importlib.metadata` fallback. See [`/version`](/core-usage/slash-commands#version).
- **Version string in startup DM** — the first-run welcome and resume message now include the running version — e.g. `hey, i'm back (ollim-bot v0.2.0). picking up where we left off.`
- **Auto-update migrated from SHA to semver tag comparison** — [`auto_update`](/configuration/reference#runtime-configuration) and [`/update`](/core-usage/slash-commands#update) now detect updates by comparing the current version against the latest `v*` tag on `origin` (`git tag -l "v*" --sort=-version:refname`). Untagged commits on `main` no longer trigger auto-updates; releases only ship on a new `v*` tag. First tagged release: **v0.1.1**.
- **Semver release workflow** — the `release.yml` GitHub Action (`workflow_dispatch` with a `patch / minor / major` bump choice) bumps `pyproject.toml`, runs checks, commits, tags, pushes, and publishes a GitHub Release — which is what `auto_update` watches for.
- **`__version__` exposed** — `ollim_bot.__version__` reads from `importlib.metadata`, with uv metadata cache invalidation fixed so version bumps show up without a clean reinstall.

## Mar 21, 2026

**Agent core changes — model auto-upgrade, bug fixes, and SDK upgrade**

- **Haiku-to-sonnet auto-upgrade** — when context exceeds haiku's 200k limit at fork creation time, the bot automatically upgrades the [interactive fork](/core-usage/forks#haiku-auto-upgrade) to sonnet. The fork embed shows a warning, and `enter_interactive_fork()` returns a `bool` indicating the upgrade. The threshold is defined in `_CONTEXT_WINDOWS` and checked against `_last_input_tokens` from `ResultMessage.usage`.

- **Compaction retry guard** — compaction retry after auto-compaction is now skipped when a fork interrupt occurred (`not fork_interrupted`), preventing a wasted API call.
- **`timestamp()` timezone fix** — `timestamp()` in `agent_context.py` now uses the `%Z` format directive to reflect the configured timezone instead of hardcoding "PT".
- **Images preserved on compaction retry** — the compaction retry query now replicates the image-aware `build_image_query()` call, preventing images from being silently dropped.
- **`set_thinking()` permission restore** — `set_thinking()` now restores `_dont_ask` from persisted config when tearing down a fork client, matching the existing fork-exit behavior.
- **Fallback text after compaction** — the `streamed` flag is now reset after compaction, so fallback `ResultMessage` text is no longer silently dropped.

## Mar 20-21, 2026

**ADHD behavior eval system and developer tooling**

  on a 1-5 scale.
- **`eval compare`** — compares latest vs previous results per scenario,
  flagging regressions when overall score drops by more than 0.3.
- **33 ADHD behavior regression tests** — 3 modules (441 lines) protecting
  prompt content, multi-step interaction flows, and context assembly. These
  run in the regular `pytest` suite without API calls.
- **Proxy prompt and goal truncation fix** — strengthened the Haiku
  user-proxy prompt for more realistic ADHD behavior, fixed qualitative goal
  truncation in `_print_scores`.
- **3 SDK docs preload skills** — developer-facing Claude Code skills
  (`/sdk-docs-ref`, `/sdk-docs-core`, `/sdk-docs-full`) with tiered context
  depth for working with Claude Agent SDK source code.

## Mar 20, 2026

**Webhook skills, subagent binding, and fire-time validation**

- **Webhook skills support** — webhooks now accept a `skills` array in YAML frontmatter, matching [routines](/scheduling/routines) and [reminders](/scheduling/reminders). Referenced skill instructions and tools are loaded at fire time. Skill tool merging moved from `scheduler.py` to `BgForkConfig.with_skill_tools()` for consistent ownership across all job types.
- **Subagent binding** — routines, reminders, and webhooks support a new `subagent` field. When set, the generated skill includes a `REQUIRED SUBAGENT` delegation instruction, and the `Agent(<name> *)` tool pattern is merged into `allowed_tools` via `BgForkConfig.with_subagent_tools()`. Unknown subagents skip execution at fire time.
- **Skill reference validation** — `validate_skill_refs()` in `skills.py` checks that every skill in a job's `skills` array has a `SKILL.md` in the skills directory. Missing refs block execution entirely — routines and reminders append a pending update with the error; webhooks log a warning and skip dispatch.
- **Webhook screening timeout** — injection screening now enforces a 30-second timeout (`_SCREENING_TIMEOUT`) via `asyncio.timeout`. If screening times out or fails, the webhook proceeds (fail-open).
- **User skills in system prompt** — `list_user_skills()` enumerates skills that are neither generated (`routine-*`, `reminder-*`, `webhook-*`) nor bundled, and injects a dynamic "Your custom skills" section into the system prompt with each skill's name and description. Inline `SKILL.md` format instructions replace the previous "search the docs" reference.

  Pre-loaded timeout and exit-strategy guidance removed from the system

  prompts and scoped to agent-initiated forks only.
- **Terminology consistency** — docstrings and internal prompts updated
  to use "background fork" consistently (docs already used correct terms).
- **Context usage warning** — a small annotation appears after responses
  when context exceeds 60% of the 200k window
  (`context: 67% (134k) — consider /compact`). Escalates at 80% with
  "compaction soon" messaging, giving you a heads-up to
  [compact proactively](/core-usage/conversations#context-and-memory)
  before auto-compaction kicks in.
- **/setup wizard** — guided `/setup` command that walks through
  configuring names, [personality](/personalizing/personality), and
  [user context](/personalizing/user-context). Detects fresh vs
  configured state and offers fast or extended paths.
- **Nested tool activity** — subagent status messages now show what tool

  a static timer.

  Discord edits for rate limit relief, updated history-reviewer for
  current `claude-history` CLI, made setup skill self-contained.
